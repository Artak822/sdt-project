# Как устроен проект

## Из чего состоит

Проект разбит на несколько независимых частей, каждая живёт в своём контейнере Docker:

- **bot** — сам Telegram-бот, с которым общается пользователь
- **user_service** — хранит пользователей и их анкеты
- **match_service** — хранит лайки и мэтчи
- **rating_service** — хранит рейтинги анкет
- **celery_worker** — фоновый процесс, пересчитывает рейтинги по расписанию
- **PostgreSQL** — база данных (одна, общая для всех сервисов)
- **Redis** — кэш (бот хранит там очередь анкет для просмотра)
- **RabbitMQ** — очередь сообщений (бот кидает туда события, сервисы их читают)

Бот общается с сервисами по HTTP. Сервисы между собой напрямую не общаются.

---

## Бот (`bot/`)

Написан на библиотеке **aiogram**. Работает в режиме polling — постоянно спрашивает у Telegram: "есть новые сообщения?".

### Как бот обрабатывает сообщения

В aiogram есть понятие **хендлер** — это функция, которая срабатывает на конкретное сообщение или нажатие кнопки. Все хендлеры разбиты по файлам:

- `handlers/start.py` — команда `/start` и кнопка "Зарегистрироваться"
- `handlers/profile.py` — создание и редактирование анкеты
- `handlers/search.py` — просмотр анкет, лайки, скипы
- `handlers/menu.py` — мэтчи, настройки, удаление профиля

### Создание анкеты (FSM)

Когда пользователь заполняет анкету, бот задаёт вопросы по очереди. Чтобы помнить, на каком шаге находится пользователь, используется **FSM (конечный автомат)** — это просто состояние, которое хранится в памяти.

Шаги: имя → возраст → пол → кого ищет → город → о себе → фото

При **редактировании** бот сначала загружает текущую анкету, сохраняет её во временное хранилище и на каждом шаге показывает текущее значение. Если нажать "↩️ Оставить как было" — значение не меняется.

### Просмотр анкет и кэш

Чтобы не дёргать базу данных на каждый свайп, бот использует **Redis** как кэш:

1. При первом запросе загружает сразу пачку анкет из `user_service` и кладёт их в Redis
2. Каждый следующий свайп берёт анкету из Redis, а не из базы
3. Кэш живёт 30 минут, потом обновляется

### События в RabbitMQ

Когда что-то важное происходит, бот отправляет событие в RabbitMQ:

- зарегистрировался пользователь → событие `user.register`
- создал анкету → событие `user.profile.complete`
- поставил лайк → событие `user.like`

Сервисы (например, `rating_service`) подписаны на эти события и реагируют на них.

---

## user_service (`services/user_service/`)

Написан на **FastAPI**. Хранит две сущности в PostgreSQL:

**Пользователь** — это Telegram-аккаунт: его ID, username, дата регистрации.

**Анкета** — это профиль для знакомств: имя, возраст, пол, город, описание, фото (хранится file_id от Telegram, не само фото).

Бот обращается к нему когда нужно создать, обновить, удалить или показать анкету, а также когда нужна лента анкет для просмотра.

---

## match_service (`services/match_service/`)

Написан на **FastAPI**. Хранит лайки и мэтчи.

Когда пользователь ставит лайк, бот отправляет запрос сюда. Сервис проверяет: а вдруг тот второй человек тоже уже лайкнул? Если да — создаётся мэтч и оба лайка помечаются как взаимные. Бот получает ответ и, если есть мэтч, сразу уведомляет обоих пользователей.

---

## rating_service (`services/rating_service/`)

Написан на **FastAPI**. Хранит рейтинг каждой анкеты.

Рейтинг состоит из трёх чисел:
- `primary_score` — базовая оценка
- `behavioral_score` — считается по активности: лайки и мэтчи
- `combined_score` — итоговый: 40% от базового + 50% от поведенческого

Сам рейтинг пересчитывает Celery Worker, а этот сервис просто отдаёт текущее значение по запросу.

---

## Celery Worker (`celery_worker/`)

Фоновый процесс, который работает по расписанию:

- **каждые 10 минут** — пересчитывает `behavioral_score` для всех анкет по формуле: `лайки × 2 + мэтчи × 5` (максимум 100)
- **каждый час** — пересчитывает `combined_score`

Использует Redis как очередь задач и работает напрямую с PostgreSQL.

---

## Как работает рейтинг

### Где считается

Рейтинг считается в двух местах:

**`celery_worker/tasks.py`** — два периодических Celery-таска:

```
recalculate_behavioral_ratings()  →  каждые 10 минут
recalculate_combined_ratings()    →  каждый час
```

Оба таска делают SQL-запрос напрямую в PostgreSQL (через синхронный SQLAlchemy) и обновляют таблицу `ratings`.

**`services/rating_service/`** — хранит и отдаёт рейтинги. Consumer слушает RabbitMQ: когда приходит событие `user.profile.complete`, создаётся начальная запись рейтинга для нового профиля с нулевыми значениями.

### Как считается

Рейтинг состоит из трёх чисел, все от 0 до 100:

**`primary_score`** — базовая оценка. Выставляется вручную (или при создании профиля через RabbitMQ-событие). Не пересчитывается автоматически.

**`behavioral_score`** — поведенческий рейтинг. Считается в `recalculate_behavioral_ratings` по SQL-запросу:

```sql
SELECT
    COUNT(DISTINCT l.id) AS likes_received,   -- сколько лайков получил профиль
    COUNT(DISTINCT m.id) AS matches_count      -- сколько мэтчей у пользователя
FROM ratings r
LEFT JOIN profiles p ON p.id = r.profile_id
LEFT JOIN likes    l ON l.to_user_id = p.user_id
LEFT JOIN matches  m ON m.user1_id = p.user_id OR m.user2_id = p.user_id
```

Формула: `behavioral = min(100, лайки × 2 + мэтчи × 5)`

Пример: 10 лайков + 3 мэтча = 20 + 15 = **35**

**`combined_score`** — итоговый рейтинг. Считается из двух предыдущих:

```
combined = min(100, primary × 0.4 + behavioral × 0.5)
```

Обрати внимание: коэффициенты дают максимум 90 (40 + 50), то есть набрать 100 можно только при `primary = 100` и `behavioral = 100` одновременно, иначе потолок ниже.

### Где применяется

**Лента анкет** — `get_feed_profiles` в `services/user_service/crud.py`.

При формировании ленты делается `LEFT JOIN` с таблицей `ratings` и сортировка по `combined_score DESC NULLS LAST` — то есть анкеты с высоким рейтингом показываются первыми, а профили без рейтинга уходят в конец.

Поскольку обе таблицы (`profiles` и `ratings`) живут в одной PostgreSQL-базе, join делается напрямую в SQL — без HTTP-запроса к `rating_service`. Ссылка на таблицу `ratings` объявлена в `crud.py` через `sqlalchemy.Table`, чтобы не импортировать модели из другого сервиса.

## Как запускать

Всё поднимается через Docker Compose. Код **встроен в образ** при сборке, поэтому после изменений нужен пересбор:

```bash
# пересобрать только бот
docker compose up --build bot -d

# пересобрать всё
docker compose up --build -d
```
